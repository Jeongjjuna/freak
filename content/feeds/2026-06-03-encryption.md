## 백엔드가 만나는 암호 기초 (2/4) — 암호화 (양방향)

> 자물쇠 금고 🔐 — 키로 잠그고, 키로 다시 엽니다.

### 실무에서 만나는 곳
- 전송 구간 암호화: HTTPS/TLS
- DB 컬럼 암호화: 주민번호·카드번호처럼 나중에 복호화해서 보여줘야 하는 값
- 설정 시크릿을 KMS/Vault로 암호화해 저장하고, 런타임에 복호화

### 포인트1 — 언제 양방향 암호화를 사용할까?
기준은 "원래 값이 나중에 필요할까?"입니다.

- 필요하다 → 암호화
- 필요하지 않다 → `해시`

주민번호, 카드번호, API Key처럼 나중에 다시 조회해야 하는 값은 암호화합니다.
반면 비밀번호는 원래 값을 알 필요가 없으므로 암호화가 아니라 `해시`를 사용합니다. (다음 편)

### 포인트2 — 암호화보다 더 중요한 것은 키 관리다
아무리 강력한 알고리즘을 사용해도 암호화 키가 유출되면 데이터는 그대로 읽힙니다.
그래서 실무에서는 알고리즘 선택보다 키를 안전하게 보관하고 접근을 통제하는 것이 더 중요합니다.
직접 구현하기보다는 KMS, Vault, 검증된 암호화 라이브러리를 사용하는 것이 일반적입니다.

<br>

대표 양방향 알고리즘 : AES(대칭키), RSA(비대칭키)

> `핵심` — 다시 읽어야 하면 양방향 암호화, 그리고 암호화를 지키는 것은 결국 키 관리입니다.
