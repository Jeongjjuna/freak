## 백엔드가 만나는 암호 기초 (4/4) — 서명

> 인감도장 🔏 — 문서에 도장을 찍어 "내가 작성했고, 중간에 바뀌지 않았다"를 증명한다.

### 실무에서 만나는 곳
- JWT 서명 (HS256, RS256)
- 웹훅 검증 (Stripe, GitHub 등)
- 서버 간 API 인증
- AWS SigV4 요청 서명

### 포인트1 — 서명은 `누가 보냈는지`와 `내용이 안 바뀌었는지`를 증명한다

해시는 내용이 바뀌었는지 확인할 수 있지만(`무결성`),
누가 만들었는지는 알 수 없다.

서명은 해시에 비밀키를 더해
"이 사람이 만들었고(`인증`), 중간에 변조되지 않았다(`무결성`)"를 증명한다.

그래서 JWT, 웹훅, 서버 간 통신에서 상대방을 신뢰할 수 있다.

### 포인트2 — 서명의 핵심도 결국 키 관리다

공격자가 서명 키를 얻으면 정상 서버처럼 서명을 만들 수 있다.

따라서 알고리즘보다 중요한 것은
키를 안전하게 보관하고 주기적으로 교체하는 것이다.

실무에서는 검증된 라이브러리를 사용하고,
키는 KMS나 Vault 같은 전용 시스템으로 관리한다.

<br>

대표 알고리즘: HMAC-SHA256(HS256), RSA(RS256), ECDSA(ES256)

> 한 줄 — 서명은 "누가 보냈는지"와 "내용이 안 바뀌었는지"를 증명한다.
