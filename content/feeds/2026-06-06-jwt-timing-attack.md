## JWT 만들다 알게 된 것 — 비밀 데이터 비교는 '상수 시간'으로!

서명, MAC, 토큰처럼 보안이 중요한 `비밀 바이트 배열(Byte Array)을 비교`할 때는 일반적인 `==`나 `contentEquals` 대신 `MessageDigest.isEqual`을 사용해야 합니다.

### 왜 그래야 할까요?
- `일반적인 비교 (`==`)`: 첫 글자(바이트)부터 비교하다가 `틀린 부분이 나오면 그 즉시 비교를 중단`합니다. 즉, `앞부분이 많이 맞을수록 비교 연산 시간이 더 오래 걸립니다.`
- `공격자의 무기`: 공격자는 토큰을 마음대로 조작해 요청을 보낼 수 있습니다. 응답 시간의 미세한 차이를 측정해 서명 값을 `한 바이트씩` 알아낼 수 있습니다. 이를 `타이밍 공격(Timing Attack)`이라고 합니다.
- `상수 시간 비교`: 맞든 틀리든 `무조건 끝까지 다 비교`한 뒤 결과를 반환합니다. 연산 시간이 항상 일정하므로, 시간 차이를 통한 정보 누출을 막을 수 있습니다.


### 한 줄 사용법
```kotlin
import java.security.MessageDigest

MessageDigest.isEqual(expected, provided) // true/false 반환, 중간에 멈추지 않음
```

### 쉽게 말하면 (문자열로)
정답이 "secret"이고, 일반적인 비교 방식을 쓴다고 가정해 볼게요.
- `"apple"` → 첫 글자부터 틀림 → 1글자만 보고 멈춤 (빠름)
- `"secXX"` → `sec`까지 맞음 → 4번째에서 멈춤 (상대적으로 느림)

공격자는 이 미세한 시간 차이를 분석해 "아, 앞 3글자는 sec가 맞구나!" 하고 확신하며 다음 글자를 찍어 맞춥니다.
상수 시간 비교를 쓰면 어떤 값을 넣든 "secret" 전체를 끝까지 비교하므로 이런 시간 단서가 완전히 사라집니다.

### 참고
