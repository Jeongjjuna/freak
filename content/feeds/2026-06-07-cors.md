## CORS (Cross-Origin Resource Sharing)

### 한 줄 정의
`프론트 JS가 다른 출처 응답을 읽어도 되는지`를 서버가 헤더로 허용해주는 규칙입니다.

### SOP가 "막는다"는 것의 의미
- `SOP(동일 출처 정책)`는 브라우저에 기본 내장된 보안 규칙입니다.
- `브라우저가, 프론트 JS가 다른 출처에서 온 응답을 읽지 못하게 차단`합니다.
- 막는 주체는 내 서버가 아니라 `사용자 브라우저`입니다. 그래서 서버끼리의 통신(curl)에는 걸리지 않습니다.
- `출처(origin)`는 스킴 + 호스트 + 포트를 말합니다. 하나라도 다르면 "다른 출처"입니다.

즉, CORS는 서버가 응답 헤더로 `이 출처는 읽어도 됨`이라고 알려줘서 SOP 차단을 풀어주는 것입니다.

### 서버는 요청자 출처를 어떻게 알까?
브라우저가 교차 출처 요청에 `Origin` 헤더를 자동으로 붙입니다.
```http
# 요청 (브라우저 → 서버)
GET /api/data
Origin: https://my-site.com

# 응답 (서버 → 브라우저)
Access-Control-Allow-Origin: https://my-site.com
```
서버는 `Origin`을 읽고, 허용 목록에 있으면 그 값을 그대로 `Allow-Origin`에 담아 되돌려줍니다(`에코`).
`*` 대신 에코하는 이유는, 자격증명 요청에는 `*`를 쓸 수 없고 여러 출처를 허용하려면 목록 검사 후 반사해야 하기 때문입니다.

### 주요 응답 헤더
```http
Access-Control-Allow-Origin: https://my-site.com
Access-Control-Allow-Methods: GET, POST
Access-Control-Allow-Headers: Content-Type
Access-Control-Allow-Credentials: true   # 쿠키 등 자격증명 허용
```

### Preflight (사전 요청)
GET/POST 같은 단순 요청이 아니라면(예: PUT/DELETE, 커스텀 헤더) 본 요청 전에 `OPTIONS`로 미리 물어봅니다. 서버가 허용 헤더로 응답해야 본 요청이 진행됩니다.

### 흔한 오해
- `차단하는 것은 브라우저`이지 서버가 아닙니다. 요청은 서버에 이미 도달했을 수도 있습니다.
- `credentials: true`면 `Allow-Origin: *`를 쓸 수 없습니다. 정확한 출처를 명시해야 합니다.
- `Origin` 헤더는 브라우저가 채우고 JS로 바꿀 수 없어서(forbidden header) 브라우저 환경에서는 신뢰할 수 있습니다. 단, curl 등 비브라우저에서는 위조가 가능하므로 CORS는 "진짜 서버 보안"이 아닙니다.

> `핵심` — 브라우저가 기본으로 다른 출처 응답 읽기를 막고(SOP), 서버가 `Origin`을 보고 허용 헤더로 알려주면 읽을 수 있게 됩니다(CORS).
