백엔드가 만나는 암호 기초 (4/4) — 서명

인감도장 🔏 — 문서에 도장을 찍어 "내가 작성했고, 중간에 바뀌지 않았다"를 증명한다.

실무에서 만나는 곳

  • JWT 서명 (HS256, RS256)
  • 웹훅 검증 (Stripe, GitHub 등)
  • 서버 간 API 인증
  • AWS SigV4 요청 서명

포인트1 — 서명은 누가 보냈는지와 내용이 안 바뀌었는지를 증명한다

해시는 내용이 바뀌었는지 확인할 수 있지만(무결성), 누가 만들었는지는 알 수 없다.

서명은 해시에 비밀키를 더해 "이 사람이 만들었고(인증), 중간에 변조되지 않았다(무결성)"를 증명한다.

그래서 JWT, 웹훅, 서버 간 통신에서 상대방을 신뢰할 수 있다.

포인트2 — 서명의 핵심도 결국 키 관리다

공격자가 서명 키를 얻으면 정상 서버처럼 서명을 만들 수 있다.

따라서 알고리즘보다 중요한 것은 키를 안전하게 보관하고 주기적으로 교체하는 것이다.

실무에서는 검증된 라이브러리를 사용하고, 키는 KMS나 Vault 같은 전용 시스템으로 관리한다.


대표 알고리즘: HMAC-SHA256(HS256), RSA(RS256), ECDSA(ES256)

한 줄 — 서명은 "누가 보냈는지"와 "내용이 안 바뀌었는지"를 증명한다.

백엔드가 만나는 암호 기초 (3/4) — 해시 (단방향)

믹서기 🥤 — 갈면 원래 재료로 되돌릴 수 없다.

실무에서 만나는 곳

  • 비밀번호 저장
  • 파일 무결성 검증
  • 캐시 키 생성
  • Git 커밋 식별자

포인트1 — 해시는 원래 값으로 되돌릴 수 없다

해시는 입력값을 고정 길이의 결과값으로 변환하는 기술이다.

같은 입력은 항상 같은 해시값을 만들지만, 해시값만 보고 원래 값을 알아내는 것은 불가능하다.

그래서 비밀번호는 원문 대신 해시값만 저장하고, 로그인 시 입력된 비밀번호를 다시 해시해서 비교한다.

포인트2 — 비밀번호는 일부러 느리게 계산한다

SHA-256은 매우 빠른 알고리즘이다.

문제는 서버뿐 아니라 공격자도 빠르게 계산할 수 있다는 점이다. 비밀번호가 유출되면 공격자는 초당 수십억 개의 후보를 대입해 볼 수 있다.

그래서 비밀번호 저장에는 bcrypt, scrypt, Argon2처럼 일부러 계산을 느리게 만든 알고리즘을 사용한다.

사용자에게는 로그인 시 0.1초 정도의 차이지만, 공격자에게는 수억 번의 시도 비용이 추가된다.


대표 알고리즘: SHA-256·SHA-512(무결성 검증), bcrypt·scrypt·Argon2(비밀번호)

한 줄 요약: 해시는 되돌릴 수 없다. 비밀번호는 느린 해시를 사용한다.

백엔드가 만나는 암호 기초 (2/4) — 암호화 (양방향)

자물쇠 금고 🔐 — 키로 잠그고, 키로 다시 연다.

실무에서 만나는 곳

  • 전송 구간 암호화: HTTPS/TLS
  • DB 컬럼 암호화: 주민번호·카드번호처럼 나중에 복호화해서 보여줘야 하는 값
  • 설정 시크릿을 KMS/Vault로 암호화해 저장하고, 런타임에 복호화

포인트1 — 언제 양방향 암호화를 사용할까?

기준은 "원래 값이 나중에 필요할까?"이다.

  • 필요하다 → 암호화
  • 필요하지 않다 → 해시

주민번호, 카드번호, API Key처럼 나중에 다시 조회해야 하는 값은 암호화한다. 반면 비밀번호는 원래 값을 알 필요가 없으므로 암호화가 아니라 해시를 사용한다.(다음편)

포인트2 — 암호화보다 더 중요한 것은 키 관리다

아무리 강력한 알고리즘을 사용해도 암호화 키가 유출되면 데이터는 그대로 읽힌다. 그래서 실무에서는 알고리즘 선택보다 키를 안전하게 보관하고 접근을 통제하는 것이 더 중요하다. 직접 구현하기보다는 KMS, Vault, 검증된 암호화 라이브러리를 사용하는 것이 일반적이다.


대표 양방향 알고리즘 : AES(대칭키), RSA(비대칭키)

한 줄 요약: 다시 읽어야 하면 양방향 암호화. 암호화를 지키는 건 키 관리.
