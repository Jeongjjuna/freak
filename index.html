불변성 (Immutability)

상태 설계 1번. 핵심은 읽기 전용 ≠ 불변.

읽기 전용인데 불변이 아니다?

List 타입은 수정 메서드만 없을 뿐, 같은 객체를 가리키는 가변 참조가 바꾸면 값도 같이 바뀐다.

val list1 = mutableListOf<Int>()
val list2: List<Int> = list1   // 읽기 전용 타입
list1.add(1)
println(list2)   // [1] ← 같이 바뀜

불변성 보장 2가지

  1. 반환 시 List 타입으로 (가변 참조를 밖에 안 넘김). 내부 리스트면 복사본:
fun getList(): List<Int> = mutableList.toList()
  1. 받은 인수는 복사해서 저장 (방어적 복사). 안 그러면 호출한 쪽이 내부 값 변경 가능:
fun setList(newList: List<Int>) { list = newList.toList() }
  • 원칙: 가변 객체 참조를 외부에 직접 내놓지 말 것. 어떤 영향 줄지 예측 불가.

부분적 불변성

함께 바뀌는 변수끼리 묶어서, "필드를 하나씩 고치기" 대신 "묶음을 통째로 새 객체로 교체"하게 만든다.

// x  가변 필드 5개가 흩어져 있음 → 하나씩 따로 바꿈
class UserDetailScreen {
    private var userName: String = ...
    private var userEmail: String = ...
    private var timeZone: TimeZone = ...
    private var onlineStatus: OnlineStatus = ...
    private var statusMessage: String = ...

    fun updateUserProfile() {
        userName = ...      // 여기서 이름만 바뀌고
        userEmail = ...     // 이메일은 아직 옛날 값인 "중간 상태"가 생김
        timeZone = ...
    }
}

// o  같이 바뀌는 것끼리 불변 묶음으로. 갱신 = 새 객체로 통째 교체
class UserDetailScreen {
    private var userProfile: UserProfile = ...   // 참조만 var
    private var userStatus: UserStatus = ...

    fun updateUserProfile() {
        userProfile = UserProfile(newName, newEmail, newTz)  // 한 번에 교체
    }
    fun updateUserOnlineStatus() {
        userStatus = UserStatus(newStatus, newMessage)
    }

    // 묶음 내부는 전부 val → 불변
    private class UserProfile(val userName: String, val userEmail: String, val timeZone: TimeZone)
    private class UserStatus(val onlineStatus: OnlineStatus, val statusMessage: String)
}

왜 좋냐:

  • 따로 관리할 가변 필드 5개 → 참조 2개로 줄어 관리 포인트 감소.
  • 함께 바뀔 값들이 항상 한 덩어리로 교체됨 → 이름만 바뀌고 이메일은 옛날 값인 어중간한 상태가 안 생김.
  • 묶음 내부는 val(불변)이라 한 번 만든 UserProfile은 절대 안 변함. 바뀌는 건 "어떤 묶음을 가리키냐"(참조)뿐.

핵심 — 읽기 전용 타입이라도 가변 참조가 살아있으면 불변 보장 X. 반환·저장은 복사로, 가변 참조는 밖에 내놓지 말 것.

CORS (Cross-Origin Resource Sharing)

한 줄 정의

프론트 JS가 다른 출처 응답을 읽어도 되는지를, 서버가 헤더로 허용해주는 규칙.

SOP가 "막는다"는 게 무슨 뜻

  • SOP(동일 출처 정책) = 브라우저에 기본 내장된 보안 규칙.
  • 내용: 브라우저가, 프론트 JS가 다른 출처에서 온 응답을 읽지 못하게 차단한다.
  • 막는 주체는 내 서버가 아니라 사용자 브라우저. 그래서 서버끼리 통신(curl)엔 안 걸린다.
  • 출처(origin) = 스킴 + 호스트 + 포트. 하나라도 다르면 "다른 출처".

→ CORS는 서버가 응답 헤더로 이 출처는 읽어도 됨이라고 알려줘서 SOP 차단을 풀어주는 것.

서버는 요청자 출처를 어떻게 아나

브라우저가 교차 출처 요청에 Origin 헤더를 자동으로 붙인다.

# 요청 (브라우저 → 서버)
GET /api/data
Origin: https://my-site.com

# 응답 (서버 → 브라우저)
Access-Control-Allow-Origin: https://my-site.com

서버는 Origin을 읽고 → 허용 목록에 있으면 → 그 값을 그대로 Allow-Origin에 담아 되돌림(에코). * 대신 에코하는 이유: 자격증명 요청은 * 불가 + 여러 출처 허용 시 목록 검사 후 반사해야 하므로.

주요 응답 헤더

Access-Control-Allow-Origin: https://my-site.com
Access-Control-Allow-Methods: GET, POST
Access-Control-Allow-Headers: Content-Type
Access-Control-Allow-Credentials: true   # 쿠키 등 자격증명 허용

Preflight (사전 요청)

GET/POST(단순) 외(예: PUT/DELETE, 커스텀 헤더)면 본 요청 전에 OPTIONS로 미리 물어봄 → 서버가 허용 헤더로 응답해야 본 요청 진행.

흔한 오해

  • 차단하는 건 브라우저, 서버 아님. 요청은 서버에 이미 도달했을 수도 있음.
  • credentials: true면 Allow-Origin: * 불가 → 정확한 출처 명시 필요.
  • Origin 헤더는 브라우저가 채우고 JS로 못 바꿈(forbidden header) → 브라우저 환경에선 신뢰 가능. 단, curl 등 비브라우저는 위조 가능 → CORS는 "진짜 서버 보안"이 아님.

핵심 — 브라우저가 기본으로 다른 출처 응답 읽기를 막고(SOP), 서버가 Origin 보고 허용 헤더로 알려주면 읽게 된다(CORS).

JWT 만들다 알게 된 것 — 비밀 데이터 비교는 '상수 시간'으로!

서명, MAC, 토큰처럼 보안이 중요한 비밀 바이트 배열(Byte Array)을 비교할 때는 일반적인 ==나 contentEquals 대신 MessageDigest.isEqual을 사용해야 합니다.

왜 그래야 할까요?

  • 일반적인 비교 (==): 첫 글자(바이트)부터 비교하다가 틀린 부분이 나오면 그 즉시 비교를 중단합니다. 즉, 앞부분이 많이 맞을수록 비교 연산 시간이 더 오래 걸립니다.
  • 공격자의 무기: 공격자는 토큰을 마음대로 조작해 요청을 보낼 수 있습니다. 응답 시간의 미세한 차이를 측정해 서명 값을 한 바이트씩 알아낼 수 있습니다. 이를 타이밍 공격(Timing Attack)이라고 합니다.
  • 상수 시간 비교: 맞든 틀리든 무조건 끝까지 다 비교한 뒤 결과를 반환합니다. 연산 시간이 항상 일정하므로, 시간 차이를 통한 정보 누출을 막을 수 있습니다.

한 줄 사용법

import java.security.MessageDigest

MessageDigest.isEqual(expected, provided) // true/false 반환, 중간에 멈추지 않음

쉽게 말하면 (문자열로)

정답이 "secret"이고, 일반적인 비교 방식을 쓴다고 가정해 볼게요.

  • "apple" → 첫 글자부터 틀림 → 1글자만 보고 멈춤 (빠름)
  • "secXX" → sec까지 맞음 → 4번째에서 멈춤 (상대적으로 느림)

공격자는 이 미세한 시간 차이를 분석해 "아, 앞 3글자는 sec가 맞구나!" 하고 확신하며 다음 글자를 찍어 맞춥니다. 상수 시간 비교를 쓰면 어떤 값을 넣든 "secret" 전체를 끝까지 비교하므로 이런 시간 단서가 완전히 사라집니다.

참고

  • 네트워크 너머 타이밍 차이는 작고 노이즈 큼 → 실제 공격은 어렵겠지만, 보안 계층을 두껍할 수 있다면 하는게 좋지 않을까? 하는 생각입니다.

핵심 — 비밀 데이터를 비교할 때는 MessageDigest.isEqual을 사용하자. "어디서 틀리든 끝까지 검사하므로, 연산 시간이 정답의 힌트가 되지 않는다."
