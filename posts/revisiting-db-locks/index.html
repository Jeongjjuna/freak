들어가며

프로젝트를 진행하다 보면 동시성 문제를 마주하게 됩니다. 실제로 장애가 발생하지 않더라도, 기능을 구현하는 과정에서 정합성 문제가 발생할 가능성은 없는지 한 번쯤 점검하게 되는데요,

저는 동시성 이슈를 검토할 때 보통 아래와 같은 단계적인 순서로 해결 방법을 고민합니다.

  1. 동시성 자체를 줄이거나 제거하는 설계 ↓
  2. DB 제약 조건 활용 (Unique Key 등) ↓
  3. DB Lock (비관적 락, 낙관적 락) ↓
  4. 비동기 순차 처리 (Queue 기반) ↓
  5. 분산 락 (Redis, Zookeeper 등)

하지만 실무에서는 구현의 편의성이나 빠른 개발에 집중한 나머지, 충분한 고민 없이 비교적 적용하기 쉬운 'Lock'을 먼저 선택하는 경우를 종종 보게 됩니다.

물론 Lock은 강력한 해결 수단이지만, 도입에 따른 데드락(Deadlock) 등 여러 장애 상황을 고려해야 하고, 특히 Redis 같은 도구를 활용한 '분산 락'의 경우 Lock 해제 실패 대응이나 운영 및 구현 복잡도까지 함께 짊어져야 합니다. 결국 시스템의 복잡도를 증가시키는 선택이 될 수 있죠.

따라서 기술을 선택할 때는 "어떤 기술을 쓰느냐"보다 "왜 이 기술이어야만 하는가"에 대한 답을 내릴 수 있어야 합니다. 그래서 저는 Lock이 필요 없는 설계가 가능한지 먼저 깊게 고민한 뒤, 아래의 기준들을 바탕으로 적절한 트레이드오프(Trade-off)를 고려해 구현 방향을 결정합니다.

  • 성능 및 비용
  • 비즈니스/서비스 정책
  • 구현 복잡도
  • 유지보수성과 협업 난이도

사람마다 우선시하는 기준은 다르겠지만, 적적한 기준을 가지고 문제에 접근하는 것이 중요하다고 생각합니다.

AI Agent 시대, 개발자에게 진짜 필요한 능력

최근 AI Agent 기반의 개발 환경을 경험하면서, 앞서 언급한 기준과 판단력의 중요성을 더욱 크게 체감하고 있습니다.

이전에는 코드를 직접 작성하는 구현 능력이 핵심이었다면, 지금은 AI에게 의도와 설계를 정확히 전달하고, 생성된 결과물이 올바르게 동작하는지 빠르게 판단·검증하는 능력이 더 중요해지고 있습니다.

AI는 순식간에 코드를 만들어 주지만, 그 코드가 올바른 동시성 모델을 따르고 있는지, 실제 운영 환경에서도 안전하게 동작할지는 결국 개발자의 책임입니다. 많이 알고 있을수록 더 잘 검증할 수 있고, 개념에 대한 깊은 이해도가 곧 개발자의 판단력이 되는 시대가 온 것입니다.

돌아보면 한 번 완벽히 이해했다고 생각했던 개념도 시간이 흐르거나 개발 환경이 바뀌면 다시 정리가 필요하곤 합니다. 그래서 이번 글에서는 초심으로 돌아가, 동시성 해결의 핵심 축인 DB Lock의 동작 방식과 선택 기준을 다시 복습해보려 합니다. 간단하게 실습을 해보고, 실제 상황에서 어떤 기준으로 기술을 선택해야 하는지 복습해보는 시간을 가지고자합니다.

동시성 문제

대부분의 서비스는 평소엔 문제 없이 동작하지만, 트래픽이 몰리는 순간 이런 문제가 드러납니다.

  • Lost Update
  • Race Condition
  • 데이터 불일치

그래서 기능을 개발할 때는 항상 "이 로직에서 동시성 문제가 발생할 수 있는가?"를 함께 고민하게 됩니다.

예를 들어 다음과 같은 질문들을 스스로 던지게 됩니다.

  • 정합성이 반드시 보장되어야 하는가?
  • 동시 요청이 많이 발생할 가능성이 있는가?
  • 실제 트래픽이 집중되는 기능인가?

실제로 동시성 이슈가 자주 발생하는 대표적인 요구사항은 다음과 같습니다.

  • 좋아요 증가
  • 주문 생성
  • 재고 감소
  • 포인트 증가/차감

이러한 문제를 해결하기 위해 데이터베이스는 Lock 메커니즘을 제공합니다.

Record Lock (Row Lock)

Record Lock은 테이블 전체가 아니라 특정 행(Row)에 대해서만 락을 거는 방식입니다.

InnoDB 스토리지 엔진에서는 UPDATE, DELETE와 같은 쓰기 작업이 수행되는 시점에 해당 레코드에 대해 Exclusive Row Lock(X Lock) 을 획득합니다.

즉, 하나의 트랜잭션이 특정 행을 수정하고 있는 동안에는 다른 트랜잭션이 동일한 레코드를 동시에 수정하지 못하도록 막아줍니다.

이를 통해 다음과 같은 문제를 방지할 수 있습니다.

  • 동일 레코드의 동시 수정 방지
  • 트랜잭션 간 충돌 제어
  • 데이터 무결성 보장

일반적으로 우리가 기대하는 "행 단위 동시성 제어" 가 바로 Record Lock이라고 볼 수 있습니다.

다만, 모든 스토리지 엔진이 Row Lock을 제공하는 것은 아니기 때문에 실제 운영 환경에서는 사용 중인 스토리지 엔진(InnoDB 여부)을 반드시 확인해야 합니다.

락 상태 확인하기

실제로 Lock이 어떻게 동작하는지 직접 확인해보는 것이 가장 확실합니다. MySQL에서는 performance_schema를 통해 현재 걸린 Lock 상태를 조회할 수 있습니다.

select *
from performance_schema.data_locks;

START TRANSACTION;

UPDATE article
SET like_count = like_count + 1
WHERE id = 123;

해당 시점에 InnoDB는 id = 123 레코드에 대해 Exclusive Row Lock(X Lock) 을 획득합니다.

performance_schema 를 통해 조회해보면 아래와 같이 record lock 이 걸렸음을 확인할 수 있습니다.

그래서 또다른 커넥션에서 아래와 같이 같은 record 에 대해 update 를 시도합니다.

START TRANSACTION;

UPDATE article
SET like_count = like_count + 1
WHERE id = 123;

이 쿼리는 즉시 실행되지 않고 대기 상태(waiting) 에 들어가게 됩니다.

이미 트랜잭션 1이 동일 레코드를 점유하고 있기 때문입니다.

트랜잭션 1이 종료되면서 해당 레코드에 걸린 Lock이 해제되고, 대기 중이던 트랜잭션 2의 UPDATE가 그제서야 수행됩니다.

이 과정을 통해 InnoDB의 Record Lock이 동일 레코드에 대한 동시 수정 요청을 원자적으로 수행한다는 것을 알 수 있습니다.

블로그에서 바로 쓰기 좋은 짧고 깔끔한 단일 콜아웃 버전으로 정리해줄게.

UPDATE 시 InnoDB의 Lock 종류는 WHERE 조건에 따라 달라집니다.

PK / Unique 조건
→ Record Lock 적용 (단일 레코드만 잠금, 동시성 영향 최소)
Non-Unique 조건
→ Gap Lock 또는 Next-Key Lock 적용 가능
→ 조회된 레코드뿐 아니라 여러 레코드 행에 INSERT까지 차단될 수 있음

DB Lock 동작은 DB 벤더, 스토리지 엔진(InnoDB 등), 트랜잭션 격리 수준에 따라 달라지므로
동시성 처리가 중요한 경우 실제 Lock 동작을 확인하며 사용하는 것이 좋습니다.

락을 오래 점유하면 생기는 문제

다만 이러한 Lock 을 오래 점유하게 되면 시스템 전체 성능에 악영향을 줄 수 있음이 예상됩니다.

  • 트랜잭션 대기 증가
    • 동일 레코드를 수정하려는 요청들이 대기 상태로 쌓인다.
    • 순간적인 트래픽 증가가 전체 요청 지연으로 이어질 수 있다.
  • DB 부하 증가
    • 대기 중인 커넥션이 늘어나면서 커넥션 풀 고갈이나 CPU 사용량 증가가 발생할 수 있다.
  • Deadlock 발생
    • 여러 트랜잭션이 서로의 Lock 해제를 기다리며 교착 상태에 빠질 수 있다.
  • 장애 상황 유발
    • Lock 대기가 길어지면 Timeout이 발생하고, 결국 서비스 장애로 이어질 가능성도 존재한다.

개발자는 Lock 을 얼마나 짧은 시간동안 점유할 수 있는가에 고민을 해야합니다.

조회수, 좋아요 기능 예시

게시글 서비스에서는 반정규화를 통해
게시글 Row에 조회수, 좋아요 수 같은 집계 데이터를 함께 저장하는 경우가 많습니다.

문제는 쓰기 트래픽이 몰리는 순간 발생합니다.

조회수 증가, 좋아요 증가, 게시글 수정은
기능적으로는 서로 독립적인 작업이지만
모두 동일한 게시글 Row를 UPDATE 하게 됩니다.

특정 게시글에 좋아요나 조회 요청이 집중되면
집계성 업데이트가 Row Lock을 지속적으로 점유하게 되고,

그 결과 게시글 수정과 같은 다른 작업까지
불필요하게 대기하는 상황이 발생할 수 있습니다.

데이터 구조 분리를 통한 Lock 경합 완화

예를 들어 다음과 같이 데이터를 분리할 수 있습니다.

article
article_like
article_like_count

락 경합을 줄이기 위해
데이터 모델의 책임을 나누는 접근입니다.

또한 조회수와 같은 집계 데이터는 요청마다 즉시 UPDATE 하지 않고, 캐시에 누적한 뒤 일정 주기로 반영하는 전략도 사용할 수 있습니다.

이번 복습을 통해 다시 느낀 점은 다음과 같습니다.

동시성 문제는 단순히 Lock 기술로 해결하는 문제가 아니라,
데이터 구조 설계 단계에서 이미 결정되는 경우가 많다는 점입니다.

비관적 락 (Pessimistic Lock)

비관적 락은 충돌이 발생할 가능성이 높다고 가정하는 방식입니다.

데이터 수정 전에 먼저 Lock을 획득하여
다른 트랜잭션의 접근을 차단합니다.

InnoDB의 Record Lock 역시 비관적 락의 대표적인 예시입니다.

방법 1 — UPDATE 한 번에 처리

UPDATE article
SET like_count = like_count + 1
WHERE id = 123;
  • UPDATE 실행 시점에 Row Lock 획득
  • Lock 점유 시간이 상대적으로 짧음
  • 불필요한 조회 과정 없음

단순 증가/감소 연산에서는 가장 좋은 방식

방법 2 — SELECT FOR UPDATE

SELECT *
FROM article
WHERE id = 123 FOR
UPDATE;

이후 비즈니스 로직 수행 후 UPDATE를 실행합니다.

  • 조회 시점부터 Lock 획득
  • 조회 → 검증 → 수정 흐름 구현 가능
  • JPA 같은 ORM 기술 사용 시 객체 중심으로 로직 작성이 용이

단점

  • Lock 점유 시간이 길어질 수 있음
  • 트랜잭션 설계에 더욱 주의 필요

낙관적 락 (Optimistic Lock)

낙관적 락은 충돌이 거의 발생하지 않는다고 가정하는 방식입니다.

데이터 조회 시 Lock을 획득하지 않고,
UPDATE 시점에 데이터 변경 여부를 검증하여 충돌을 감지합니다.

동작 방식

  • 현재 version 조회
  • UPDATE 시 version 조건 포함
  • 수정 성공 시 version 증가
UPDATE article
SET like_count = ?,
    version    = version + 1
WHERE id = 123
  AND version = 10;
  • UPDATE 성공 → 충돌 없음
  • UPDATE 실패 → 다른 트랜잭션이 먼저 수정

충돌이 발생한 경우에는 재시도(Retry) 전략을 함께 설계해야 합니다.

특징

  • DB Lock 대기 없음
  • 동시 처리 성능에 유리
  • 충돌 발생 시 애플리케이션 레벨에서 처리 필요 적합한 상황
  • 충돌 빈도가 낮은 서비스
  • 읽기 비중이 높은 시스템
  • Lock 대기를 최소화하고 싶은 경우

마무리

동시성 문제를 해결하기 위해 DB Lock을 고려할 때,
무조건 Lock부터 적용하기보다는 설계를 통해 문제 자체를 줄일 수 있는지 먼저 고민하는 것이 중요합니다.

또한 충돌 발생 빈도가 낮거나 일부 실패를 허용할 수 있는 경우라면,
제약조건이나 원자적 연산을 활용해 가능한 한 Lock을 사용하지 않는 방향을 먼저 검토해보는게 좋은 방향이라는 생각이 듭니다.

이번에 정리하며 스스로 정립한 사고 흐름은 다음과 같습니다.

  • 데이터 구조를 분리할 수 있는가?
  • 원자적 UPDATE로 해결 가능한가?
  • Lock이 반드시 필요한 상황인가?
  • 비동기 처리로 문제 자체를 단순화할 수 있는가?
    (단, 운영 비용과 서비스 정책에 대한 협의가 필요)
  • 복잡한 비즈니스 제어나 다중 인스턴스 환경이라면 분산 락을 고려할 수 있는가?
