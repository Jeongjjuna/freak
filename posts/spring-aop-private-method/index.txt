1f:T18a3,<h2 id="1-개요">1. 개요</h2>
<p>사내에서 Spring AOP를 활용한 FeatureFlag 도입을 맡게 되었다. 커스텀 어노테이션과 Spring AOP를 조합해 메서드 단위로 기능을 ON/OFF하거나 분기 처리할 수 있는 구조를 설계했다.</p>
<p>사용 방식은 간단하다. <code>@FeatureFlag</code> 어노테이션이 붙은 메서드가 호출될 때 AOP가 개입하여, 설정값에 따라 해당 메서드를 실행할지, 아니면 대체 메서드로 분기할지를 결정한다.</p>
<p>그런데 사용 중 한 가지 문제를 발견했다. <strong>private 메서드에는 AOP가 동작하지 않는 것이다.</strong> 처음엔 구현 실수인가 싶었지만, 조사해보니 이건 Spring AOP의 근본적인 동작 방식에서 비롯된 제약이었다.</p>
<h2 id="2-proxy-란">2. Proxy 란?</h2>
<p>Spring AOP는 <strong>프록시(Proxy) 패턴</strong>을 기반으로 동작한다. 스프링 빈을 생성할 때, AOP가 적용된 클래스는 원본 객체 대신 <strong>프록시 객체</strong>가 빈으로 등록된다.</p>
<p>프록시는 대리인 역할을 하는 객체로, 원본 객체의 앞단에서 요청을 가로채 부가 로직을 실행한 뒤, 원본 메서드를 호출한다.</p>

<h2 id="3-spring-aop에서-private-메서드를-가로챌-수-없는-이유">3. Spring AOP에서 private 메서드를 가로챌 수 없는 이유</h2>
<p>Spring AOP는 기본적으로 <strong>JDK 동적 프록시</strong> 또는 <strong>CGLIB 프록시</strong> 방식을 사용한다.</p>
<ul>
<li><strong>JDK 동적 프록시</strong>: 인터페이스 기반 프록시 생성</li>
<li><strong>CGLIB</strong>: 클래스 상속 기반 프록시 생성</li>
</ul>
<p>두 방식 모두 <code>private</code> 메서드를 오버라이딩하거나 가로챌 수 없다.</p>
<p>Java에서 <code>private</code> 메서드는 <strong>상속되지 않으며</strong>, 외부에서 접근이 불가능하다. 따라서 프록시 객체가 이 메서드를 오버라이딩하는 것 자체가 언어 스펙상 불가능하다.</p>
<h2 id="4-transactional-도">4. @Transactional 도?</h2>
<p><code>@Transactional</code>도 Spring AOP 기반이기 때문에 동일한 제약이 있다.</p>
