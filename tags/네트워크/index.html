#네트워크태그 전체 태그

피드

(1)
freak·1개월 전·2026.06.01 20:36

시스템을 구축하다 보면 마주치게 되는 server to server 구간의 HTTP 통신. 이때 서버 B는 들어온 요청이 정말 서버 A로부터 온 것인지 신뢰할 수 있어야 한다.

주로 JWT나 HMAC 같은 서명(인증) 방식으로 요청의 신뢰성을 증명한다. "내용이 위변조됐는지" / "누가 보냈는지"를 검증하는 것이다. 내용 자체의 기밀성은 HTTPS(암호화)가 담당한다.

  1. JWT 방식

보내는 서버 A가 JWT에 서명하고, 받는 서버 B가 그 서명을 검증하여 약속된 클라이언트가 맞는지 확인한다. 대칭키(HS256)를 쓰면 서버 간에 시크릿 키를 공유하고, 비대칭키(RS256 등)를 쓰면 A가 개인키로 서명하고 B는 공개키로 검증하므로 키를 공유하지 않는다.

  1. HMAC (Hash-based Message Authentication Code)

서버 간에 시크릿 키를 공유한다. 요청 하는 서버는 요청 정보(Request Body 등)를 시크릿 키를 이용해 해시 기반 서명값(MAC)으로 변환하고, 그 값을 "헤더에 담아 전송"한다. 요청을 받은 서버는 요청 정보(Request Body 등) 을 동일한 시크릿 키로 해시 기반 서명값(MAC)으로 변환하고, "그 결과가 헤더의 값과 같은지 확인하여 검증"한다.

HMAC을 사용하면 시크릿 키가 유출되지 않는 한, 중간에서 공격자가 요청 정보를 위변조하더라도 유효한 서명값을 만들 수 없어 탐지·거부된다. 그래서 보안이 중요한 금융권이나 웹훅 검증에서 주로 사용된다. 다만 HMAC 서명만으로는 정상 요청을 그대로 가로채 다시 보내는 재전송(replay) 공격은 막지 못한다. timestamp와 nonce를 함께 서명에 포함하면 이를 보완할 수 있다.

정리

신뢰할 수 있는 내부망 네트워크 안에서의 server to server 통신이라면 JWT 방식으로도 충분하다. "요청 정보가 변조되지 않았는지", "정말 시크릿 키를 가진 소유자가 보낸 요청이 맞는지"까지 검증해야 한다면 HMAC을 사용한다.
