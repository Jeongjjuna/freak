백엔드가 만나는 암호 기초 (4/4) — 서명

인감도장 🔏 — 문서에 도장을 찍어 "내가 작성했고, 중간에 바뀌지 않았다"를 증명한다.

실무에서 만나는 곳

  • JWT 서명 (HS256, RS256)
  • 웹훅 검증 (Stripe, GitHub 등)
  • 서버 간 API 인증
  • AWS SigV4 요청 서명

포인트1 — 서명은 누가 보냈는지와 내용이 안 바뀌었는지를 증명한다

해시는 내용이 바뀌었는지 확인할 수 있지만(무결성), 누가 만들었는지는 알 수 없다.

서명은 해시에 비밀키를 더해 "이 사람이 만들었고(인증), 중간에 변조되지 않았다(무결성)"를 증명한다.

그래서 JWT, 웹훅, 서버 간 통신에서 상대방을 신뢰할 수 있다.

포인트2 — 서명의 핵심도 결국 키 관리다

공격자가 서명 키를 얻으면 정상 서버처럼 서명을 만들 수 있다.

따라서 알고리즘보다 중요한 것은 키를 안전하게 보관하고 주기적으로 교체하는 것이다.

실무에서는 검증된 라이브러리를 사용하고, 키는 KMS나 Vault 같은 전용 시스템으로 관리한다.


대표 알고리즘: HMAC-SHA256(HS256), RSA(RS256), ECDSA(ES256)

한 줄 — 서명은 "누가 보냈는지"와 "내용이 안 바뀌었는지"를 증명한다.

인코딩 · 암호화 · 해시 · 서명 — 한 번에 구분하기

비슷해 보이지만 목적과 방향성으로 나누면 명확해진다.

구분목적방향성키대표 알고리즘
인코딩 (Encoding)형식 변환 (보안 아님)양방향 (키 불필요)xBase64, Base64URL, Hex
양방향 암호화 (Encryption)기밀성 — 내용 숨기기양방향 (복호화 가능)oAES(대칭키), RSA(비대칭키)
해시 (Hash)무결성 — 데이터 지문(위/변조 검사)단방향 (복원 불가)xSHA-256, SHA-512, SHA-3
서명 (Signature)무결성 + 인증 — 위변조 + 출처단방향 (해시 기반)oHMAC-SHA256(HS256), RS256, ES256

🤔 자주 헷갈리는 포인트

인코딩 ≠ 암호화

  • Base64는 키가 없다.
  • 누구나 디코딩할 수 있다.
  • 시크릿을 Base64로 저장하는 것은 보안이 아니라 단순한 형식 변환이다.

해시 ≠ 서명

  • 해시는 데이터의 지문을 생성한다.
  • 같은 입력은 항상 같은 해시값을 만든다.
  • 데이터가 변경되면 해시값도 달라진다.

반면 서명은

  • 해시에 키를 결합하여 생성한다.
  • 데이터가 변경되지 않았음을 검증할 수 있다.
  • 누가 생성했는지까지 검증할 수 있다.

즉,

서명(Signature) = 해시(Hash) + 키(Key)

라고 생각하면 이해하기 쉽다.

JWT의 HS256은 암호화가 아니다

JWT의 HS256은 HMAC-SHA256을 사용하는 서명 알고리즘이다.

토큰 내용을 숨기는 것이 아니라,

  • 토큰이 위조되지 않았는지
  • 신뢰할 수 있는 발급자가 만들었는지

를 검증하기 위해 사용된다.

한 줄 요약

형식만 바꾸면 인코딩, 숨기면 암호화, 지문(위/변조검사) 찍으면 해시, 지문(위/변조검사)에 도장까지 찍으면 서명.
