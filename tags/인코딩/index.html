백엔드가 만나는 암호 기초 (1/4) — 인코딩

한국어를 영어로 번역하는 것 🌍— 내용은 그대로, 표현만 바뀐다.

실무에서 만나는 곳

  • JWT의 header, payload (Base64URL)
  • URL의 한글·특수문자 (Percent-Encoding)
  • 바이너리 데이터를 JSON에 담을 때 (Base64)
  • 바이너리를 로그·DB에 저장할 때

왜 사용할까?

시스템마다 다룰 수 있는 데이터 형식이 다르기 때문이다.

예를 들어 Base64는 바이너리를 텍스트로 바꿔 JSON, HTTP, DB 등에 담을 수 있게 해주고, Percent-Encoding은 한글·특수문자를 URL에 안전하게 담을 수 있게 해준다.

포인트1 — JWT는 암호화된 토큰이 아니다

JWT의 header와 payload는 Base64URL로 인코딩된 데이터일 뿐이다. 누구나 디코딩해서 내용을 볼 수 있으며, JWT의 안전성은 인코딩이 아니라 서명(Signature) 에서 나온다.

포인트2 — 인코딩은 보안이 아니다

인코딩의 목적은 데이터를 숨기는 것이 아니라 전달과 저장이다. 참고로 Base64는 데이터 크기를 약 33% 증가시킨다.


대표 인코딩: Base64, Base64URL, Hex, Percent-Encoding

한 줄 요약: 인코딩은 보안 기술이 아니라 데이터를 표현하는 방법이다.

비슷해 보이지만 목적과 방향성으로 나누면 명확해진다.

구분목적방향성키대표 알고리즘
인코딩 (Encoding)형식 변환 (보안 아님)양방향 (키 불필요)xBase64, Base64URL, Hex
양방향 암호화 (Encryption)기밀성 — 내용 숨기기양방향 (복호화 가능)oAES(대칭키), RSA(비대칭키)

서명 (Signature)무결성 + 인증 — 위변조 + 출처단방향 (해시 기반)oHMAC-SHA256(HS256), RS256, ES256

🤔 자주 헷갈리는 포인트

인코딩 ≠ 암호화

  • Base64는 키가 없다.
  • 누구나 디코딩할 수 있다.
  • 시크릿을 Base64로 저장하는 것은 보안이 아니라 단순한 형식 변환이다.

해시 ≠ 서명

  • 해시는 데이터의 지문을 생성한다.
  • 같은 입력은 항상 같은 해시값을 만든다.
  • 데이터가 변경되면 해시값도 달라진다.

반면 서명은

  • 해시에 키를 결합하여 생성한다.
  • 데이터가 변경되지 않았음을 검증할 수 있다.
  • 누가 생성했는지까지 검증할 수 있다.

즉,

서명(Signature) = 해시(Hash) + 키(Key)

라고 생각하면 이해하기 쉽다.

JWT의 HS256은 암호화가 아니다

JWT의 HS256은 HMAC-SHA256을 사용하는 서명 알고리즘이다.

토큰 내용을 숨기는 것이 아니라,

  • 토큰이 위조되지 않았는지
  • 신뢰할 수 있는 발급자가 만들었는지

를 검증하기 위해 사용된다.

한 줄 요약

형식만 바꾸면 인코딩, 숨기면 암호화, 지문(위/변조검사) 찍으면 해시, 지문(위/변조검사)에 도장까지 찍으면 서명.
